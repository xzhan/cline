Policy-Bound Agent Context Spec

用户登录后，Hub 根据身份、场景、workspace 和策略生成 Agent 可见且 runtime 强制执行的上下文。

Hub central policy Human-in-loop MCP tools File scopes Linux whitelist
Core idea: Agent Context 不是 prompt 文本补充，而是一个双层对象： model-visible context 告诉模型能做什么； runtime-enforced policy 在 MCP、文件、shell、审批层真正拦截动作。

1. Requirement Summary

after login

Functional Requirements

  • 登录后构建 per-user, per-session Agent Context。
  • 按场景注入 MCP tool list、file access、Linux command whitelist。
  • 支持 VS Code、Hub、CLI、GChat/Slack connector、EDA Linux server 等场景差异。
  • 支持用户申请自定义目录 scope，但必须经过 Hub admin policy 约束。
  • 支持对敏感 command 的 human-in-loop confirmation。
  • 所有 tool/file/command action 都必须 fail closed。

Non-Goals

  • 不让 prompt instruction 单独承担安全边界。
  • 不允许 human approval 绕过明确 deny policy。
  • 不把 EDA tool 语义硬编码到通用 agent loop。
  • 不默认开放任意 shell，例如 bash -c、rm -rf、任意脚本执行。

Model Where config lives Best for Risk
Central Hub Admin Hub database/config service Enterprise control, audit, EDA license governance Needs Hub availability and admin UI
Workspace Repo Config .cline/, .clinerules/, repo policy files Project-specific paths, scripts, and local rules Can drift per repo unless locked or signed
Layered Model Hub baseline + workspace rules + scenario + approval Long-term production architecture Needs clear precedence and conflict UI

Precedence Rule

Global deny > Hub admin policy > workspace policy > scenario policy > human approval

Human approval can approve only an action that is already allowed but marked approval_required.

Context Builder Inputs

  • User identity and role.
  • Session source: Hub, CLI, VS Code, connector.
  • Workspace root and repo policy.
  • Hub admin policy and global deny list.
  • Runtime capabilities and available MCP servers.

Runtime Enforcement Points

  • MCP tool registry filters tool exposure.
  • File tool canonicalizes paths and checks scopes.
  • Shell tool checks command profile and file effects.
  • Approval service gates sensitive actions.
  • Audit service records decisions and outcomes.

6. Command + File Combined Effect

preflight algorithm

Command Profile

{
  "command": "make vcs TEST=*",
  "cwd": "/proj/chip/top",
  "reads": [
    "/proj/chip/top/rtl/**",
    "/proj/chip/top/tb/**",
    "/proj/chip/top/scripts/**"
  ],
  "writes": [
    "/proj/chip/top/sim/**",
    "/proj/chip/top/logs/**",
    "/proj/chip/top/waves/**"
  ],
  "requiresApproval": true,
  "reason": "uses VCS license and writes simulation artifacts"
}

Runtime Check

  1. Canonicalize command, cwd, paths, and symlinks.
  2. Check command against whitelist and denied patterns.
  3. Expand command profile reads/writes/deletes.
  4. Intersect effects with effective file policy.
  5. Block if any effect touches deny path.
  6. Ask approval when policy says approval required.
  7. Execute only exact approved command/effect tuple.
effective permission = user policy ∩ workspace policy ∩ command profile effects
